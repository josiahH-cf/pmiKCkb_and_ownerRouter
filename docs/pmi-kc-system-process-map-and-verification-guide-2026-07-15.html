Reading rule

What this document claims—and what it does not

Claims are grounded in current page components, server routes, data modules, tests, Tier-0 facts, and current production evidence. “Implemented” means code exists; “reachable” means a page or API path exposes it; “activated” means the deployed configuration permits it. Those are intentionally not treated as synonyms.

Where a process begins

A question, source publication, Process start, lease review, typed/voice maintenance report, tokenized public maintenance submission, Gmail workflow link, or Admin/Vendor identity action.

✅ Confirmed

What the app owns

Workflow and Test state, app tickets, decisions, approvals, source metadata, links, notification state, policies, bodyless receipts, and audit trails in Firestore.

✅ Confirmed

What external tools own

Gmail owns message bodies, RentVine owns property/lease/work-order records, the renewal Sheet owns its worklog, and Drive owns files. External changes require a separate exact action.

Reachability finding The codebase contains typed S20 external orchestration and Live Vendor service modules, but ordinary Lease/Maintenance pages do not invoke the S20 external executor, and no Live Vendor invite/OAuth/Gmail routes or controls are exposed. Most non-Gmail Live actions are therefore visible as readiness contracts—not currently operable UI actions. This distinction is a confirmed implementation fact and a high-priority review target.

✅ Flow E — Decision and external-action authorization

Approval authorizes a prepared action; it does not itself perform the external effect. The executor must still win the one-attempt claim and prove the result.

  1. The server creates an immutable preview from trusted context: actor, lane, action, target, effect, dependencies, and server-owned risk.
  2. A human exactly confirms Low/Medium action context, or an authorized Approver/Admin handles High-risk approval with matching preview/context hashes.
  3. The Action Registry and per-action contract must permit the current lane. A visible “ready” matrix alone is not execution.
  4. An atomic claim permits one attempt. A concurrent or duplicate request returns existing state and makes no second provider call.
  5. The executor records a receipt and provider readback; uncertain delivery becomes an ambiguity/reconciliation state.

System-wide invariant tests

Mode isolation

Precondition: matching Live and Test-looking records. Action: exercise reads/actions. Pass: badges remain explicit, Test stores cannot construct providers, and Live failures never substitute Test data.

Exact-effect confirmation

Precondition: prepared action. Action: mutate target/context, expire, reuse, or race confirmation. Pass: zero second provider calls; stale inputs fail before construction.

Source-state honesty

Precondition: canonical, conflicting, open, missing, and excluded sources. Action: ask the same question. Pass: only eligible citations support an answer; uncertainty stays visible.

Role and scope

Precondition: Editor, Approver, Admin, restricted-scope staff, Vendor. Action: direct-navigate and invoke APIs. Pass: server results agree with visible navigation and record projection.

Atomic evidence

Precondition: transition that writes primary state and audit/activity. Action: force a write failure/race. Pass: either all required records commit or none do; duplicates return existing state.

Content minimization

Precondition: Gmail/Vendor message operation. Action: inspect DOM, Firestore records, audit, logs. Pass: app records remain bodyless and bounded; secrets/tokens/content do not leak.

7 · System-wide interpretation

✅ Confirmed

One Next.js application coordinates three product lanes: knowledge/operations, Lease Renewal, and Maintenance with Workflow Communications. Internal staff and external Vendors cross different authentication and authorization boundaries.
