Production Release and Live Test Guide

A practical owner-and-operator guide to the functionality now merged into the PMI KC KB, how to deploy it to the existing production Cloud Run service, and how to decide whether each feature works as intended.

Prepared July 11, 2026 Target: pmi-kc-kb-prod Service: pmi-kc-kb-demo Human-controlled sends and writes
Release boundary: deployment, live smoke tests, Gmail mailbox access, and external writes are owner-gated. This release does not authorize autonomous sending or a RentVine, Sheet, QuickBooks, bank, or other system-of-record write.

2. Before deploying

  • Use a pmikcmetro.com identity only.
  • Pull the merged main branch and confirm the worktree is clean.
  • Run session auth in the owner's interactive terminal.
  • Confirm .env.production.local contains reviewed production values and no placeholders.
  • Confirm the approved source manifest is outside git at temp/client-production-source-manifest.json.
  • Run the budget guard, production preflight, and consolidated cutover report.
  • Review the current Cloud Run revision before deployment so rollback is immediate. 
git switch main
git pull --ff-only origin main
npm install
npm run auth:session
npm run preflight:adc
npm run check:budget-guard
npm run preflight:production -- --env-file=.env.production.local
npm run cutover:report -- --manifest=temp/client-production-source-manifest.json --env-file=.env.production.local --json
A production-valid notification configuration can leave the documented Gmail-send residual in the cutover report. Do not bypass it with an allow-notifications flag unless a separate, explicit send approval is granted.

3. Deploy the merged release

Deploy Firestore rules and indexes first, then update the existing Cloud Run service: 

npm exec firebase -- deploy --only firestore:rules,firestore:indexes --project pmi-kc-kb-prod

npm run deploy -- --project=pmi-kc-kb-prod --service=pmi-kc-kb-demo --region=us-central1 --search-location=us --budget-confirmed --allow-multiple-spaces --service-account=pmi-kc-kb-runtime@pmi-kc-kb-prod.iam.gserviceaccount.com

Canonical production URL: https://pmi-kc-kb-demo-kq6wuvpiva-uc.a.run.app

Stop immediately if the deploy targets another project or identity, the budget guard fails, the preflight reports a new blocker, or the canonical URL changes unexpectedly.

4. Automated production smoke

Establish the signed-in browser profile once, then reuse it for the Ask and Dictate smokes: 

$baseUrl = "https://pmi-kc-kb-demo-kq6wuvpiva-uc.a.run.app"

npm run smoke:auth-live -- --base-url=$baseUrl --pause-on-human
npm run smoke:ask-live -- --base-url=$baseUrl --browser-session
npm run smoke:transcribe-live -- --base-url=$baseUrl --browser-session

Pass when sign-in returns to the app, Ask returns a grounded answer with source behavior intact, and Dictate returns HTTP 200 through the real speech boundary.

5. Feature-by-feature live acceptance

A. Mobile renewal decider

  1. Sign in as an Admin and open /lease-renewal/live on a phone-sized viewport.
  2. Open the decider mode and confirm exactly one renewal flag card is visible with “1 of N” progress.
  3. For an approved Low/Medium test flag with a suggested winner, choose “Accept suggested source.”
  4. Confirm the decision advances without a free-text prompt and the audit reason reads “Accepted the suggested source.”
  5. Confirm “Approve write-back” is a distinct follow-on action and does not claim the change was written.
  6. Use Skip and confirm the card stays deferred during that browser session without creating a resolution.
  7. For a High/Blocked or manually corrected test flag, confirm a plain-English reason is required.

Pass: one-card flow, correct conditional reasons, durable audit, distinct authorization, and no external write.

B. Scoped sub-user access

  1. In /admin/users, choose an approved internal test user and assign the Maintenance scope with a reason.
  2. Have that user sign out and back in so the refreshed claim is active.
  3. Confirm Maintenance remains visible and usable.
  4. Confirm Lease Renewals and Approval Queue are absent or redirect to the user's primary in-scope surface.
  5. Confirm the Admin audit records the scope change and the user's role did not change.
  6. Restore the intended scope after the test and record the reason.

Pass: the scope subtracts access without granting a role or exposing an out-of-scope route.

C. Unified Console and Notifications

  1. Open the Console and record the “Needs your decision” count.
  2. Open /notifications from the notification menu.
  3. Confirm the hub includes the same decision signal plus connection and Space-coverage conditions.
  4. Confirm values such as addresses, proposed values, and decision reasons are not exposed in summary rows.
  5. Use “mark all read” and confirm event notifications clear while standing setup conditions remain until fixed.
  6. As an Admin, confirm the team-review digest appears as one value-free rollup rather than one alert per edit.

Pass: Console and hub agree, the hub is a superset, summaries remain value-free, and no email is sent.

D. Gmail Hub

  1. Open /gmail-hub and confirm draft, template/triage, and thread-summary tools render together.
  2. Paste sanitized test facts into the triage/template workspace and confirm a suggested label and draft preview appear.
  3. Paste a sanitized test thread and confirm a structured summary, waiting-on party, and suggested next action appear.
  4. Test an excluded category such as legal/notices and confirm the system refuses before AI drafting.
  5. Confirm every mailbox-read affordance says “Waiting on Gmail access.”
  6. Confirm there is no Send control and no claim that the inbox was read.
  7. Optionally, with the approved compose-only runtime configured, prepare one renewal draft and confirm it appears as UNSENT in the approved mailbox. Delete the test draft afterward.

Pass: pasted-content tools work, exclusions fail closed, mailbox read stays gated, and any created message remains an unsent draft.

E. Anticipated work

  1. Open the Console and find the Anticipated work lane.
  2. Confirm renewal, owner-outreach, and notice items are computed from the existing planners.
  3. Confirm unsupported Maintenance/compliance families show an honest waiting/no-source state.
  4. Confirm the caption states that work is computed on request and never scheduled or sent.
  5. Use a start link only on an approved test case and confirm it opens a test-run or desk flow rather than silently starting production work.

Pass: real supported projections, honest unsupported states, and no scheduler or autonomous start.

F. Approval Queue and maintenance supporting improvements

  1. Confirm Approval Queue opens to one urgent-first “Needs your decision” list with alternate views under disclosure.
  2. On an eligible safe Low/Medium queue row, confirm inline approval records the app-plane decision.
  3. Confirm High-risk, self-assigned, renewal-flag, and write-back rows still require their full surfaces.
  4. In Maintenance, confirm unit search suggests live RentVine units without auto-selecting an uncertain match.
  5. Confirm assignment shows approved internal users, supports “Assigned to me,” and never exposes a raw UID as the normal label.

Pass: faster triage and assignment without weakening risk, actor, or system-of-record gates.

6. Release-wide safety checks

  • No control sends an email automatically.
  • No renewal approval changes RentVine or the operating Sheet.
  • Gmail Hub never claims to read the mailbox.
  • The only executable registry key is the compose-only unsent renewal-draft action.
  • A maintenance-only user cannot access renewals or Approval Queue.
  • Summary cards do not expose addresses, proposed values, free-text reasons, or customer records.
  • Unsupported or missing sources render uncertainty instead of generic property-management advice.
