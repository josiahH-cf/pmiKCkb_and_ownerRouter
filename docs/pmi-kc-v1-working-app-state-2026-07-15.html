External Vendor V1 walkthrough

Vendor authentication is part of V1. The canonical Test Vendor uses the real Firebase password/TOTP flow and assigned-ticket authorization, while its mailbox is app-only and rejected before OAuth or Gmail client construction. Its stable Vendor record and invented work survive an authentication reset; the replaceable Firebase UID and every credential bound to it do not.

Admin previews the exact identity change

The fixed alias is vendor:test-summit-plumbing at service@summit-plumbing.example.invalid. The Admin supplies a plain-English reason and reviews the effect.

Admin confirms once

Firebase creates the Test user, claims bind vendor_id and data_mode:test, and the setup link is returned only in the confirmed Admin response. If that response closes before use, Admin can exact-preview a replacement only for the same reconciled pending_setup Test identity. Links are no-store, never stored, emailed, or written to audit.

Vendor sets a password and enrolls TOTP

The first sign-in enrolls an authenticator factor. The app then signs out deliberately; enrollment alone is not treated as an authenticated Vendor session.

Vendor signs in again with password + TOTP

The server requires a fresh sign-in whose Firebase token records TOTP as the second factor before issuing the one-hour Vendor session.

Vendor sees only the assigned Test ticket

Vendor mode, assignment mode, and ticket mode must match. Unassigned and cross-Vendor tickets remain invisible and inaccessible.

Vendor operates the app-only Test mailbox

Read the ticket thread, form a reply proposal, apply approved Waiting/Complete labels, review the exact reply, confirm once, and receive an idempotent non-external receipt.

Admin disables and revokes

The Admin previews the exact account, confirms with a reason, disables the Firebase principal, and revokes active sessions. If a reset is already claimed or prepared, disable stops before Firebase or audit—even after the reset lease expires—so the Admin recovers that reset first. A completed reset does not prevent a later disable.

Admin can reset and safely start again

From pending setup, active, or disabled, an Admin supplies a reason and reviews the bound status/invite-version transition. Confirmation deletes the old Firebase principal, creates a new UID, increments the invite version, returns the stable Vendor record to pending_setup, and returns one new no-store setup link. A concurrent or stale confirmation fails closed and cannot disable a reset another Admin just completed. If the browser reloads after the app record reached prepared, the normal Admin preview binds the original reset generation without displaying its UID. While the two-minute owner lease is live, re-entering the original reason returns the same confirmation and a different reason or takeover refuses. After expiry, the Admin may enter a fresh reason; the server safely rebinds only that reason to the validated original lifecycle and records the takeover without requiring manual database repair. If disable finished before reset claimed the lifecycle record, the old active-state confirmation is rejected; a fresh disabled-state preview can reset normally.

Stable work, replaceable authentication

Reset preserves vendor:test-summit-plumbing, its assigned Test tickets, app-only mailbox history, labels, completed receipts, and current assignment state. It invalidates the old Firebase UID, password, TOTP factors, sessions, password-action links, and UID-bound activation or reply confirmations. Removed assignments stay removed; reset never silently reassigns work.

Test Vendor production configuration

Firebase Email/Password is enabled, the project-wide MFA state and TOTP provider are both enabled, and the deployed host is authorized. No Gmail OAuth, DWD, routable mailbox, or token vault is required for the Test principal.

An expired reset never inherits abandoned credentials

Recovery quarantines the abandoned Firebase generation by disabling it, clearing factors, revoking sessions, and deleting it—even when its claims look exactly right. The winner must receive a fresh UID distinct from the original source, the staged Firestore record, and the Auth user found at the Test address. A prepared recovery keeps the original single invite-version increment and canonical reset audit. It adds a separate bodyless recovery-claim event—who recovered, which Test Vendor, a hash of the fresh reason, and when—so recovery is visible without being mistaken for a second reset. If the old request finishes late, it cannot mint a second link, mark the winner complete, disable it, or delete it.

Audit shows attempts and outcomes honestly

Reset and setup-link recovery record a bodyless claim event when one Admin wins the lifecycle lease, then a separate completion event only after the state change succeeds. A failed post-claim attempt therefore leaves an honest claim trail without falsely claiming completion. Audit stores actor, Test Vendor, reason hash, action, and time—never the plaintext reason, Firebase UID, password, setup link, TOTP material, or mailbox content.

Revocation closes reads and writes

Every Test mailbox read, draft/label change, confirmation, and reply commit rechecks the current active Vendor UID, active Test assignment, ticket/thread join, and reset state in the same transaction. Disable, deassignment, UID rotation, or a claimed/prepared reset therefore stops stale reads and writes before mailbox state or a receipt can change.

Optional Live Vendor activation adds

A real routable Vendor mailbox, the Vendor's own consent, same-address server-side Google OAuth, the exact approved Gmail scopes, redirect URI, and Secret Manager refresh-token vault. Those inputs activate Live Gmail for that Vendor; they are not prerequisites for the required working Test authentication/reset lifecycle.

Human acceptance still to record Pending — the canonical Test Vendor must finish its one-time password setup, enroll TOTP, sign in again with the fresh second factor, and confirm assigned-ticket-only access in the production browser. The production check also exercises disable → reset/re-enable → fresh setup so UID rotation and preserved Test work are proven without exposing secrets. Automated UI acceptance separately forces a prepared interruption, reloads the Admin page, confirms that only the original reason reproduces the UID-free preview while the lease is live, then uses a fresh reason after expiry to prove the fresh-UID takeover and its separate bodyless recovery audit. The secret-bearing step necessarily stays with the human holding the password and authenticator; no password, setup key, reset link, or TOTP code belongs in this report or repository.

What the earlier verifier language meant

Both quoted updates were accurate snapshots of an earlier release process. Their terminology was easy to misread, and the old governance model over-counted repeated evidence rows.

“All pre-wrapper gates are green: formatting, typecheck, lint (0 errors; 8 known warnings), 286 unit files/1,984 tests, Firestore 16/56, core E2E 32 passed/18 intentionally skipped, cutover rehearsal fully green, and the release report correctly remains Pre‑V1 with 169 open external gates. Dependency audit is limited to three Moderate dev-only firebase-tools-chain findings already held for named disposition.”
“Pre-wrapper gates”
The checks run individually before the final all-in-one shell wrapper had passed. It did not mean the production deployment or signed-in browser walkthrough was complete.
Formatting / typecheck / lint
Source formatting had no drift, strict TypeScript compiled, and lint had zero errors. The eight warnings were known non-blocking warnings at that commit—not eight failures.
286 files / 1,984 tests
That historical commit passed 1,984 unit assertions distributed across 286 test files. The final release-candidate result is larger and is reported separately as 306 files / 2,178 tests.
Firestore 16/56
This meant 16 Firestore test files and 56 passing checks. It was not “16 passed out of 56,” nor a 29% score. The later full local-candidate baseline is 17 files / 59 tests.
32 passed / 18 skipped
Thirty-two core end-to-end scenarios passed. Eighteen were intentionally conditional on an emulator, Live account, or another named prerequisite; they were visible skips, not hidden failures.
Cutover rehearsal green
A synthetic production-like manifest generated safe import/deploy/rollback plans and rejected malformed input. A rehearsal was never a claim that a deployment or rollback occurred.
“Pre-V1”
The old report required every external-evidence row and named acceptance to close before it would label the whole app V1. That definition has been superseded by working-app V1 plus independent provider activation.
169 external gates
These were checklist/evidence rows, not 169 missing features, screens, bugs, or client decisions. Most repeated the same provider-proof requirements across many typed actions.
Three Moderate findings
All three traced to one development-only Firebase CLI dependency chain. No High or Critical finding appeared, and the affected packages were absent from the production runtime tree. “Held for named disposition” means the owner, bounded rationale, trigger conditions, and August 15 recheck are documented; it does not mean the finding was ignored.
