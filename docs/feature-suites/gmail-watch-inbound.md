# S31 - Gmail reply-watch continuous activation + operator follow-up

> New 2026-07-23 (operator note): authored from `docs/roadmap-unblock-2026-07-23.md` feature #5 (Wave 2, §3 `D-AUTOMATION-LINE`, §5 owner-dependency #3); governed by `Q-GMAIL-WATCH-OWNER` (amended 2026-07-23) and `F-ROADMAP-BUILD-AUTHORIZED` in `docs/facts.md`.

**Goal.** Today the app already notices when an owner or tenant replies on a linked renewal or maintenance thread and raises it as attention, but the underlying Gmail watch is not continuously active: a Gmail watch expires roughly every seven days and only a manual, human-confirmed renewal re-arms it, so inbound pickup silently lapses; and there is no surface that tells an operator "we emailed this owner and have heard nothing back in N days." After S31 a Cloud Scheduler job keeps the read-only watch armed automatically so replies keep flowing into the existing attention surface without anyone remembering to renew, and the Renewal and Maintenance desk shows which linked threads have gone quiet and lets an operator open a threaded follow-up draft that a human reviews and sends. Nothing is ever sent automatically: the scheduler only renews a read-only watch and raises attention, and every client-facing follow-up stays operator-initiated and exact-confirmed by a person.

**What it is / how it functions.** The inbound half is already built and stays untouched in behavior; S31 adds continuous renewal at the provider seam and an operator-facing quiet-thread and follow-up surface in the app plane. The existing mechanism this suite builds on:

- **Inbound push, already built (`app/api/gmail-hub/pubsub/route.ts` and `processGmailPushNotification` in `lib/gmail-hub/service.ts`).** Pub/Sub delivers a Gmail history notification; the handler authenticates the Pub/Sub service identity first (`verifyPubSubPushRequest`), requires the `gmail.mailbox.read` gate, walks bounded history, matches new messages to LINKED workflow threads, and calls `markCommunicationAttention`, which flips the bodyless `WorkflowCommunicationLink` to `attention_required` and appends a `reply_attention_created` audit. It never replies or sends.
- **Watch mutation, already built (`app/api/gmail-hub/watch/route.ts` and `watchMailbox` in `lib/gmail-hub/service.ts`).** A human `edit`-capable operator previews (`GET`, `watchPreview`) then confirms (`POST`, `confirmed:true`, must match the exact server preview so `mailboxEmail === user.email` and `topicName === config.topicName`); the service claims a one-attempt key, calls `client.watchMailbox(topicName)`, reads back the Gmail-assigned expiration, and is ambiguous-on-failure. It is read-only and, per its own preview copy, "does not send a message or grant cross-mailbox access."
- **Attention surfacing, already built (`lib/gmail-hub/notifications.ts`).** `listGmailWorkflowNotifications` projects `attention_required` links into the unified notifications hub (renewal and maintenance families, decision lane), space-scoped and retention-filtered.
- **Reply threading, already built (`buildOutgoingPayload` in `lib/gmail-hub/service.ts`).** Any workflow reply sets `inReplyTo = parent.messageId` and `references = dedupe(parent.references, parent.messageId)` capped at 20, so a follow-up stays in the same Gmail thread; a generic `kind:"new"` compose is refused with 409.

The gap is narrow: nothing renews the watch on a schedule (it silently expires), and a link that reached `status:"sent"` and never received a reply is invisible (no quiet-for-N-days surface, no follow-up entry point).

- **Buildable now (app-plane).**
  - **No-reply selector (new `lib/gmail-hub/follow-up.ts`).** A pure read over the existing bodyless `listCommunicationLinks` projection: a linked outbound thread is awaiting-reply when `status === "sent"`, no inbound attention has landed (`attention_at_ms` is unset and status is not `attention_required`), and `now - updated_at_ms >= followUpAfterDays`. It reads only fields already persisted on `WorkflowCommunicationLink` (`status`, `attention_at_ms`, `updated_at_ms`, `gmail_thread_id`), touches no message body, adds no new external scope, and stays `production_allowed:false`. When a reply lands the push path flips the same link to `attention_required`, so it leaves the awaiting-reply set automatically.
  - **Quiet-thread surface (extend `lib/gmail-hub/notifications.ts` and `components/gmail-hub/WorkflowCommunicationPanel.tsx`).** Surface the awaiting-reply links as a distinct "No reply in N days" operator prompt beside the existing `attention_required` items, space-scoped and read-markable the same way, so an operator sees quiet threads without opening Gmail.
  - **Operator-triggered follow-up entry point (extend `components/gmail-hub/WorkflowCommunicationPanel.tsx`).** On a quiet thread the operator clicks "Draft follow-up"; the panel opens the existing prepare, confirm, and send reply flow pre-scoped to that `gmail_thread_id` and workflow purpose, so the follow-up reuses `buildOutgoingPayload` threading and the existing exact-confirm human send. S31 mints no new send or draft action key: it reuses the already-governed draft and reply keys, and a human initiates, edits, and exact-confirms every send. There is no scheduled or autonomous creation of a client-facing draft.
- **Build to the seam (live provider).**
  - **Scheduler-invoked read-only auto-renew endpoint (new `app/api/gmail-hub/watch/renew/route.ts` plus a service method that reuses `watchMailbox`).** A server-to-server POST that authenticates the Cloud Scheduler service identity by OIDC exactly as `verifyPubSubPushRequest` authenticates the Pub/Sub push identity (bearer OIDC token, verified email, `email_verified` true, an allowed `@pmi-kc-kb-prod.iam.gserviceaccount.com` service account, expected audience equal to this endpoint HTTPS URL). It does not use the interactive `confirmed:true` preview-match, which is the human UI contract; instead the renewal is gated by the same `gmail.mailbox.read` action gate, a renew-only-when-near-expiry guard (renew only when `currentWatchExpirationMs` is inside a lead window, otherwise no-op), and the same one-attempt idempotent `watchMailbox` claim with Gmail expiration read-back. It calls the identical read-only watch mutation, so it performs no send and grants no new scope.
  - **Config and inertness (extend `lib/gmail-hub/pubsub.ts` with a sibling `readGmailWatchSchedulerConfig`).** The endpoint reads the scheduler service account and audience from env; when that config is absent it returns 503 "not configured" (mirroring `readGmailPushConfig`), so the built endpoint is inert until the owner provisions the topic and job. The watch topic is the same `projects/pmi-kc-kb-prod/topics/...` the push path already requires.
  - **Bodyless renewal audit and the full one-attempt, idempotent, receipted, reversible contract.** Each auto-renew records a bodyless audit (mailbox key, topic hash, prior and next expiration, attempt outcome) reusing the existing watch-attempt store; an ambiguous outcome consumes the attempt key and does not retry, matching the human path. Reversibility: removing the Scheduler job or the configured watch stops future renewal and delivery, and a later renewal replaces the expiration.
- **Owner dependency (the one flip).** Exactly `docs/roadmap-unblock-2026-07-23.md` §5 item 3: the owner creates (1) the Gmail-watch Pub/Sub topic `projects/pmi-kc-kb-prod/topics/<gmail-watch>` with the Gmail push service account granted publish and the authenticated push subscription pointed at `/api/gmail-hub/pubsub`, and (2) a Cloud Scheduler job that POSTs the auto-renew endpoint on a cadence inside the Gmail watch lifetime (roughly seven days, so for example daily) using an OIDC service identity the endpoint trusts. Both are read-only: the topic carries inbound notifications and the job only re-arms the read-only watch. After the owner step the flip is the standard one-line reviewed change from roadmap §6: set the scheduler config env so the endpoint is no longer inert, and, if a distinct auto-renew action key is registered rather than reusing `gmail.mailbox.read`, set its `readiness`, `evidence_status`, and `production_allowed`, add it to both `EXECUTABLE_ALLOWLIST` copies (`scripts/seed-action-registry.ts` and `lib/admin/migration-readiness.ts`), and update the pinned schema tests. No client-facing send gate is touched by this flip.

**Open questions & assumptions.** This authoring pass creates only the spec file, so the `Q-`/`A-` rows below are recorded here and must be added to `docs/facts.md` "## Open Questions" by the build slice that picks this up.

- _Assumption:_ the follow-up window default `followUpAfterDays` is 3 business days, offered as confirm-with-default to the owner; it is an operator setting, not a client or vendor decision. If the owner prefers another interval the selector reads it from config with no behavior change.
- _Assumption:_ `gmail.mailbox.read` is already executable for the live push path (the built inbound watch and push-to-attention flow depend on it). If it is not executable in the target lane, enabling the read gate precedes auto-renew activation and follows the same roadmap §6 recipe.
- _Assumption:_ the follow-up reuses the existing draft and reply action keys and their gates; S31 introduces no new client-facing send or draft key and changes no `production_allowed` flag for a client-facing send.
- _Open:_ whether the scheduler auto-renew is a distinct action key (for example `gmail.watch.auto_renew`) or reuses the `gmail.mailbox.read` gate directly. Default is to reuse the read gate and register a distinct key only if the per-endpoint allowlist model requires one.
- _Assumption:_ single configured operator mailbox (single-Workspace build per roadmap §7); the scheduler renews the configured mailbox, not a per-tenant watch.
- _Client-owned:_ none. The owner (not the client) provisions the Pub/Sub topic and Cloud Scheduler job; there is no irreducible client, vendor, or legal decision, so no `docs/client-checklist.md` row is required.

**Forbidden actions / hard gates.** A violation of any of these is itself a falsification (roadmap §7):

- Pushes raise attention only. The push handler never auto-replies or auto-sends; its render-attention path has no send call, and that must remain true (AC-S31-2).
- Follow-ups are operator-triggered. There is no scheduled or autonomous creation of a client-facing draft, per `D-AUTOMATION-LINE`; a human initiates and exact-confirms every client-facing send.
- The watch renewal is read-only. It calls only `watchMailbox` (no send, no cross-mailbox scope), and the scheduler drives no send. Manual renewal remains the fallback until the job is armed.
- Generic non-workflow `gmail.message.send` stays Registry-closed, and a generic `kind:"new"` compose stays refused.
- The personal `josiah.abernathy@gmail.com` account never enters the watch or auth path; the mailbox and service identities are the `pmikcmetro.com` mailbox and `@pmi-kc-kb-prod` service accounts.
- No secrets, customer PII, or guessed provider endpoints enter git or evidence; the topic, service account, and audience come from env or Secret Manager.
- The roughly ten dollar total cost ceiling holds; the Scheduler cadence stays inside the watch lifetime so there is no runaway polling.
- Every live effect is one-attempt, idempotent, receipted, and reversible; deploys, the Pub/Sub topic, the Scheduler job, and any scope grant are owner-run.
- S31 may build the live auto-renew provider path to the seam and prepare its inert enablement, but it does not set any client-facing send `production_allowed:true`. The read-only auto-renew activates only when the owner provisions the topic and job and the scheduler env is set, and any new action key flip updates both `EXECUTABLE_ALLOWLIST` copies plus the pinned schema tests.
