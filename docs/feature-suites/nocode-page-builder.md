# S37 - Full no-code page/layout builder

> New 2026-07-23 (operator note). Implements owner decision D-BUILDER-FULL (roadmap-unblock-2026-07-23 feature #11, Â§3, Â§4 Wave 3) under `F-ROADMAP-BUILD-AUTHORIZED`. Multi-slice suite; pure app-plane, with no live provider and no owner flip.

**Goal.** Today, changing the app's pages or adding a new one means an engineer hand-edits React/Next.js; only content (SOPs, templates, tools, placeholders, rules, roles, and process definitions) is editable inside the app. After S37 an Admin can build and rearrange app pages with no code: create a new page, add, reorder, and remove sections drawn from a fixed library of pre-built display components, preview the result, and publish it live through the same Draft-to-Active validation and immutable-version lifecycle that S21 already runs for process definitions. A published page is inert presentation: it displays existing app data and links to existing in-app surfaces, and it can never grant execution, enable an external action, change the Action Registry, connector policy, environment config, or system prompt, or widen a role or scope. The end-state is that the app's own pages and layout become an Admin-editable, versioned, reversible, audited artifact, with zero engineer involvement for a layout change and zero new external authority created.

**What it is / how it functions.**

- **Page and section schema - `lib/page-builder/schema.ts` (new).** A Zod `PageDefinition`: `id`, `slug` (URL-safe, unique, namespaced under `/pages/`), `title`, optional `space_id` (Space scope), `required_capability` (default `read`; may be `manageAdmin` for an admin-only page), `status` (`Draft`/`Active`/`Retired`, mirroring the process-definition status set), and an ordered `sections: SectionInstance[]`. Each `SectionInstance` is `{ id, component_type, props, visibility? }` where `component_type` is an enum drawn only from the section library and `props` is a typed, bounded, per-component record of plain display data (text, an in-app route link, or a reference to an existing read-only data source id). No prop may be an Action Registry key, an external-host URL that triggers a side effect, or a raw HTML, script, or iframe string.
- **Section library - `lib/page-builder/section-library.ts` (new).** A fixed, code-owned registry of pre-built, side-effect-free presentational components (heading, inert markdown/rich-text block, in-app link list, Space or process summary card, KB-answer embed that reuses the existing read path, image reference to an already-validated publication image, callout, divider, single two-column layout). Each entry declares its `component_type`, a Zod prop schema, and a render function. The component types an author may use are exactly this registry; an unknown type is rejected at validation and rendered as a safe placeholder, never executed. This mirrors the process action-reference allowlist in `components/workflows/ProcessDefinitionDetailClient.tsx`: authored content picks from pre-registered building blocks and cannot introduce new executable ones.
- **Renderer - `components/page-builder/PageRenderer.tsx` (new) + `app/pages/[slug]/page.tsx` (new).** A server component resolves the Active page definition by slug (via the publication Active pointer), enforces `required_capability` through `requirePageCapability` / `requirePageRole` (`lib/auth/page-guards.ts`) and Space scope through `hasSpaceAccess` (`lib/auth/session`), wraps output in the existing `AppShell` (`components/layout/AppShell.tsx`), and maps each `SectionInstance` to its library component. The renderer reads props as data only; it imports no executor, calls no action gate, and issues no external request. Data-bearing sections call existing read-only loaders (for example the Space/process summary reuses the `getProcessDefinition` path). An unknown or unvalidated section renders an inert "Unavailable section" placeholder.
- **Persistence and editable layer - `lib/firestore/page-definitions.ts` (new).** `create` / `update` / `reorder` / `softDelete` mirror `lib/firestore/editable.ts` exactly: `assertCan("edit")` plus a `manageAdmin` gate, `assertWritableSpace` (Space scope and `read_only` boundary), a transaction that writes the doc, and an append-only `change_log` entry (`createChangeLog`, `entity_type` `page`). Draft edits stay mutable; only publish freezes a version. Slug uniqueness is enforced in-transaction like `assertUniqueTemplateName`.
- **Editor - `components/page-builder/PageBuilderClient.tsx` (new).** Mirrors the `ProcessDefinitionDetailClient` pattern (Save and Publish buttons, a publication note, a status pill, a `fieldset` disabled unless `canEdit && canManageAdmin && status not in {Retired}`). The Admin adds a section by picking a `component_type` from the library, edits its props via the component's typed prop form, reorders with move-up/move-down over the ordered array, removes a section, and creates a new page (slug, title, scope, `required_capability`). Save PATCHes the draft; Publish POSTs to the publish route.
- **Publish (Draft-to-Active) lifecycle - `lib/publication/page-definition.ts` (new).** A near-exact analog of `lib/publication/process-definition.ts`: serialize the definition snapshot to JSON and call `publishTrustedContent` (`lib/publication/service.ts`) with `resourceType: "page_definition"`, producing an immutable publication version, an atomically updated Active pointer, an append-only `publication_audit`, and rollback via `rollbackTrustedPublication`. The `extendCommit` hook writes a `page_definition_versions` doc and flips the page's `active_version_id`, exactly like the process-definition path. A page carries no `action_references` and no `processActionKeys`.
- **Validation extension - `lib/publication/validation.ts` + `lib/publication/authority-firewall.ts`.** Add a `page_definition` branch to `resolveAllowedType` (`application/json`, a conservative cap of 512 KiB) and a page-shape check analogous to `assertProcessShape`: sections present, section ids unique, every `component_type` in the section library, every prop passing its component's Zod schema and length bounds, and `assertAuthorityFieldsAreInert(parsed)` run over the whole definition so any structured attempt to smuggle `action_registry`, `claims`, `connector_policy`, `enable_action`, `executor`, `production_allowed`, `role_scope`, or `system_prompt` is refused with `authority_field_forbidden`. Because a page carries no action keys, `assertRegisteredProcessActions([])` is trivially satisfied and a page can never reference an action key at all. Extend `assertRollbackAllowedByCurrentPolicy` with a `pageDefinitionAllowed` case beside the existing `processDefinitionAllowed` one so a rolled-back page is rechecked against the current policy.
- **Admin surface and optional nav - `app/admin/pages/**`(new) +`components/layout/AppShell.tsx`/`components/layout/PrimaryNav.tsx`.** The builder index lives under `/admin`(Admin-only). A published page may optionally be Admin-curated into the primary nav, still capability and scope filtered exactly as today; the curated entry can only target an in-app`/pages/<slug>` route and is itself an Admin page-config change, never author-driven.

- **Buildable now (app-plane).** Everything in this suite. The schema, the section library, the renderer and dynamic route, the editable-layer persistence with `change_log` plus scope and `read_only`, the Admin editor, the reuse of the publish, validation, immutable-version, and rollback machinery, the authority-firewall extension, the `manageAdmin` gating, and the optional curated nav. It adds no system-of-record write, no autonomous send, and no new external scope; a page definition performs no external effect and stays outside the Action Registry entirely, so `production_allowed` is not even a concept for it. The loop builds all of it unattended across the three phases below and stays `production_allowed:false` because it registers no action.
- **Build to the seam (live provider).** None. This suite has no external provider, no third-party API, no credential, and no live effect to build toward. A page definition is inert app-plane configuration; there is nothing to stub with a fake provider and nothing to replace with a live one.
- **Owner dependency (the one flip).** None. There is no external endpoint, credential, scope grant, vendor confirmation, or billing approval to activate. The suite ships entirely on the app plane. The only human gate is the ordinary in-app Admin (`manageAdmin`) who authors and publishes pages, which is a product capability rather than an owner console step. The routine `npm run deploy` that ships every merged change is the owner's normal step, not a feature-specific dependency.

**Open questions & assumptions.**

- _Answered 2026-07-23 (D-BUILDER-FULL):_ the owner chose the full schema-driven page/layout builder (page model plus renderer plus editor), not content-only editing; this suite is that build.
- _Assumption:_ page definitions are `data_mode` live app configuration (the real chrome), but they are excluded from the Action Registry and carry no `processActionKeys`, so no `production_allowed` gate applies; they are safe by construction because the section library is a fixed code-owned allowlist.
- _Assumption:_ V1 reordering is an ordered array with move-up and move-down controls; free-form drag-and-drop and nested grids beyond the single two-column layout section are a later slice, not V1.
- _Assumption:_ the markdown/rich-text section renders as inert text (no raw HTML, no script, no iframe), and links are restricted to same-origin in-app routes; external links, embeds, and arbitrary HTML are out of scope for V1 because they would reintroduce an injection and off-app-navigation surface.
- _Assumption:_ a page cannot become or shadow the sign-in, admin role-management, vendor, or any existing hard-coded security route; slugs are namespaced under `/pages/` and a collision with an existing route is rejected at create time.
- _Open:_ whether a non-Admin Editor may ever author pages within their own Space. Current decision is no (`manageAdmin` only, matching the hard gate); revisit only on an explicit owner decision. Record as `Q-PAGEBUILDER-EDITOR` in `docs/facts.md` if raised. (Not added at authoring time because this task creates only the spec file; the ordered sequence below records it when the suite runs.)

**Adversarial acceptance checks.**

Phase 1 - page model and renderer:

- **AC-S37-1** - A persisted `PageDefinition` with an ordered section list round-trips through create, update, reorder, and soft-delete with one append-only `change_log` entry per mutation and Space-scope plus `read_only` enforcement; a scope-restricted principal, a read-only Space, or a non-Admin caller is refused 403 and writes nothing. _Verify:_ `npm test -- page-definitions`; `npm run test:firestore`.
- **AC-S37-2** - `GET /pages/<slug>` renders the Active definition inside `AppShell`, maps each section to its library component, and renders an inert "Unavailable section" placeholder for an unknown or unvalidated `component_type`; the rendered route imports no executor and issues no external request (no action-gate call, no provider fetch). _Verify:_ `npm test -- page-renderer`; keep `action-gate` green.

Phase 2 - Admin editor and publish/validation/version lifecycle:

- **AC-S37-3** - The Admin builder (`manageAdmin`) can add, reorder, remove sections, and create a new page; the same routes return 403 for an Editor or Approver, and for an Admin acting outside an assigned Space, persisting nothing. _Verify:_ `npm test -- page-builder route-auth-boundary`.
- **AC-S37-4** - Publish runs the full validation pipeline and, only on pass, writes an immutable `page_definition` version, flips the Active pointer, and appends a `publication_audit`; an oversize body, a wrong MIME, a page graph with duplicate section ids or an unknown `component_type`, or a scanner-unavailable outage creates no Active pointer and returns a specific safe error. Rollback creates a new audited version that reuses a prior validated content reference without deleting later history and refuses a target the current policy no longer allows. _Verify:_ `npm test -- page-definition publication-validation publication-versioning`; `npm run test:firestore`.

Phase 3 - authority firewall and scope/nav containment:

- **AC-S37-5** - A page definition carrying structured `action_registry`, `claims`, `connector_policy`, `enable_action`, `executor`, `production_allowed`, `role_scope`, or `system_prompt` fields is refused with `authority_field_forbidden` and creates no version; a published page references zero Action Registry keys and cannot make any disabled action executable, change config or system prompt, or widen a role or scope. _Verify:_ `npm test -- publication-authority-firewall page-definition action-gate`.
- **AC-S37-6** - A published page respects its `required_capability` and Space scope: a user without the capability or Space access is redirected or denied exactly as the existing page guards do, and a curated nav entry for the page is capability and scope filtered and can only target an in-app `/pages/<slug>` route with no external href. _Verify:_ `npm test -- page-renderer route-auth-boundary`.
- **AC-S37-7** - Full checks pass. _Verify:_ `npm run format:check`, `npm run lint`, `npm run typecheck`, `npm test`, `npm run verify:spec-traceability`, `npm run build`; keep `feature-suite-spec-shape.test.mjs` green.

**Forbidden actions / hard gates.** Admin-only: every authoring, reorder, delete, and publish route requires `manageAdmin`; an Editor or Approver cannot author pages. A page or section can NEVER grant execution, enable or register an external action, change the Action Registry, connector policy, environment config, or system prompt, or widen a role or scope; this mirrors the S21 authority firewall (`assertAuthorityFieldsAreInert` over `FORBIDDEN_AUTHORITY_FIELDS`), and a page carries no `processActionKeys` and touches no executor. The renderer never invokes the action gate or executor and issues no external effect; layout never bypasses the action gate because a page cannot reference an action to gate. Published pages are validated exactly like process definitions (type `application/json`, size cap, MIME detection, malware, sensitivity, and page-graph validity), failing closed on any unavailable required scanner. The section library is a fixed code-owned allowlist; an author cannot introduce a new component type, raw HTML, script, iframe, or external link or embed, and rich text renders inert. Slugs cannot shadow the sign-in, `/admin/*`, or vendor routes; pages live under `/pages/`. Standard safety NEVERs are preserved (roadmap Â§7): no autonomous client-facing send (internal-staff notifications may auto-send per `D-AUTOMATION-LINE`, but this suite sends nothing); generic non-workflow `gmail.message.send` stays Registry-closed; the personal `josiah.abernathy@gmail.com` account never enters an auth path; no secrets, PII, or guessed endpoint in git; ~$10 budget cap; and deploys stay owner-run. Suite-specific hard stop: because this suite creates no external action, it never sets `production_allowed:true`, never touches either `EXECUTABLE_ALLOWLIST` copy, and never edits the action-registry schema tests; if a slice would need any of those, it has left the page-builder boundary and must stop.
