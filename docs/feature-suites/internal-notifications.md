# S39 - Internal transactional notifications + in-app notification center

> New 2026-07-23 (operator note). Implements owner decision D-AUTOMATION-LINE (roadmap-unblock-2026-07-23 §3, scope rows #13 + #5): internal-staff notifications may auto-send, every client-facing send stays human-confirmed. Wave-1, pure app-plane, zero external owner dependency. EXTENDS S17 (`unified-console-and-attention.md`); it reuses that hub and never forks it. Disposable packet: `docs/temp/internal-notifications-plan.md` (local-only).

**Goal.** Today staff learn that feedback was filed only by opening the Admin feedback panel by hand, and the owner-configured transactional destination is saved to Firestore but wired to nothing (`TransactionalDestinationPanel` literally says "Today it is display-only"). End state, both halves internal-only: (1) the S17 `/notifications` hub gains an Admin-scoped Feedback section that shows, value-free, how many reports were just filed and how many are past their follow-up window, drawn from ONE gather so the hub and the `/admin` feedback panel can never show different numbers; and (2) when a report is filed the app auto-sends one internal transactional email to the owner-configured internal destination (internal staff only), because D-AUTOMATION-LINE authorizes internal-staff automation. No client, tenant, owner-of-record, or vendor ever receives an automated send from this suite, and every client-facing renewal/maintenance send stays human exact-confirmed exactly as it is today. Every claim traces to roadmap §2 row 13 / §3 D-AUTOMATION-LINE, the S17 spec, `F-SUPP-1`, or is labeled an assumption.

**What it is / how it functions.** The suite EXTENDS the one S17 attention machinery rather than adding a parallel one: the same value-free attention contract (`lib/attention/lanes.ts`), the same pure feed builder (`lib/notifications/feed.ts`), the same server assembly (`lib/notifications/hub.ts`), and the same single-gather interlock discipline (one source of truth so counts never diverge). Feedback is NOT a "needs your decision" item, so it is a new lane fed by its own gather, never folded into `gatherNeedsDecisionInbox` (which would wrongly inflate the decision count on every surface). Modules named below are the real ones read for this spec.

- **The pieces we reuse.** `support_reports` is the existing Admin-reviewable feedback queue (`lib/firestore/support-reports.ts`, `F-SUPP-1`); `SupportReportStatus` is `new | acknowledged | resolved` (`lib/firestore/types.ts:453`). `app/api/report-issue/route.ts` files a report to that queue and returns `delivered` honestly, and its comment already reserves an email `subject` as "forward-compat for the TIX-6 email subject". The owner transactional destination is an Admin-editable single Firestore doc (`lib/firestore/owner-transactional-destination.ts`) seeded to `DEFAULT_OWNER_TRANSACTIONAL_EMAIL`, edited by `components/admin/TransactionalDestinationPanel.tsx` through `app/api/admin/transactional-destination/route.ts`. TODAY its schema (`UpdateOwnerTransactionalDestinationInputSchema`, `lib/firestore/schemas.ts`) validates only `.email()` with NO domain check, and `readOwnerTransactionalDestination(actor)` asserts Admin (403 for a non-Admin caller), so this suite MUST (a) add an internal-domain allowlist (for example `@pmikcmetro.com`) to that schema so an Admin cannot save an external destination, and (b) resolve the destination for auto-send through a non-actor-gated SYSTEM read (the emit fires for every authenticated reporter, most of whom are not Admin). Both are required for the recipient lock to be a code fact, not a convention. Both the feedback panel (`SupportReportsPanel`) and the destination panel already mount on `app/admin/page.tsx`.

- **Buildable now (app-plane).**
  - **B1 - the `support` attention lane.** Add `support` to the closed `ATTENTION_LANES` enum plus its `ATTENTION_LANE_META` entry (`lib/attention/lanes.ts`); the meta `Record<AttentionLane, ...>` makes the new entry compile-enforced. All-clear copy stays positive per `verify:copy-voice` (for example "All feedback has been followed up."). No value-bearing key is added; the six-key `AttentionSignal` shape is unchanged.
  - **B2 - the single support gather.** A new pure, Admin-scoped `gatherSupportAttention(user)` (`lib/attention/support-lane.ts`, mirroring `lib/attention/standing-signals.ts` and `lib/attention/review-lane.ts`) reads only the app's own `support_reports` collection and returns value-free signals in the `support` lane: one "N new feedback filed" signal and one "M follow-up due" signal, where follow-up-due is computed on read from status + age (no new Firestore index, no new collection). Signals carry counts + lane + a deep link to the `/admin` feedback panel only; never a description, element hint, route value, or reporter identity. This ONE gather feeds both the hub and the `/admin` panel badge, so their numbers cannot diverge (extends the S17 single-gather interlock).
  - **B3 - the `support_reports` notification family, Admin-gated at serve time.** Add `support_reports` to `NOTIFICATION_FAMILY_KEYS` + `NOTIFICATION_FAMILIES` mapped to the `support` lane (`lib/notifications/families.ts`), available and muteable, NOT a `NotificationSource` (it is a standing condition with no `read_at`, like connections/coverage/team_review, so mark-read does not apply). This moves the S17 available-family count from seven to eight; the S17 family-count invariant is EXTENDED to eight, not weakened.
  - **B4 - wire it into the one hub.** Extend `BuildNotificationFeedInput` with a `support?: readonly AttentionSignal[]` channel filtered by the same mute / lane-threshold / snooze low-alarm resolver, returned as `feed.support` (`lib/notifications/feed.ts`); and in `loadNotificationHub` gather it only when `full && isAdmin` and filter the `support_reports` family to Admin at serve time, exactly mirroring the existing `team_review` Admin gate (`lib/notifications/hub.ts:102,127`). The S17 `/notifications` page (`app/notifications/page.tsx`) renders the Admin Feedback section. Support is standing and Admin-scoped, so it does NOT count toward the bell's unread-EVENT badge (consistent with S17's treatment of connections/coverage/team_review).
  - **B5 - the `/admin` panel badge from the same gather.** `components/admin/SupportReportsPanel.tsx` shows its "new" and "follow-up due" counts from `gatherSupportAttention` output, so the Admin panel and the hub are one number by construction (AC-S39-2).

- **Build to the seam (live provider).** The "live provider" here is the real internal transactional email transport that replaces the unused display-only destination scaffold. It is internal-only and MAY auto-send per D-AUTOMATION-LINE, so the S25/S26-style contract is honored with the human-confirm step replaced by the automated internal authorization plus an idempotency guard and a recorded receipt.
  - **Executor with a hard, code-enforced recipient lock.** `lib/notifications/internal-transactional.ts` resolves its recipient ONLY from a non-actor-gated SYSTEM read of the owner transactional destination (NOT `readOwnerTransactionalDestination(actor)`, which asserts Admin and would 403 the Editor/Viewer reporters who file most feedback); a caller-supplied recipient is ignored; an absent or blank destination fails closed (refuse, never guess); and the resolved destination MUST pass the internal-domain allowlist (enforced both when an Admin SETS it in `UpdateOwnerTransactionalDestinationInputSchema` AND re-asserted here at send time) or the send is refused. It NEVER addresses a tenant, owner-of-record, vendor, or free-form/external address.
  - **Metadata-only payload.** The body carries route pathname, origin, reporter role, ISO timestamp, and a deep link to `/admin` only. It never carries the free-text description or element hint, preserving the `F-SUPP-1` / TIX-8 invariant that the description lives only inside the Admin-gated queue and Cloud Logging sees metadata only.
  - **Dedicated Action Registry key, not generic send.** A new key `internal.transactional_notice.send` seeded `production_allowed:false` (`scripts/seed-action-registry.ts` seed plus its `EXECUTABLE_ALLOWLIST` twin in `lib/admin/migration-readiness.ts`). It is a distinct narrow key, so the generic `gmail.message.send` blast path stays Registry-closed and untouched. Transport reuses the already-approved Gmail send scope through the existing internal transactional sender identity; no NEW external scope or credential is introduced.
  - **Preview / one-attempt / receipt / rollback.** The executor computes a target-labeled preview (recipient + payload hash), sends one attempt per dedup key (`support_report:{id}:filed`), reads back and records a receipt with `delivered:true|false`, and surfaces failures honestly through a health projection mirroring `ApprovalQueueNotificationHealth`. Reversibility is the gate-off kill switch plus the no-double-send idempotency guard plus the receipt; a filed report is emitted from `app/api/report-issue/route.ts` AFTER the queue write, so a send failure never blocks the durable queue write.
  - **Test lane first.** Prove recipient-lock, metadata-only, idempotency, and honest delivery in the isolated Test lane with the gate OFF before any flip.

- **Owner dependency (the one flip).** NONE external. Unlike the Wave-2 seam suites (S30/S31/S34/S35/S36), S39 has no documented-endpoint, credential, scope grant, vendor confirmation, or billing approval to wait on: internal-staff auto-send is already authorized by D-AUTOMATION-LINE, the destination is an app-plane owner-configured Admin field, and delivery uses the already-approved Gmail send scope. The gate flip to `production_allowed:true` (the roadmap §6 recipe: seed `readiness`/`evidence_status`/`production_allowed` + both `EXECUTABLE_ALLOWLIST` copies + the pinned `action-registry-schema.test.ts` and `seed-action-registry-allowlist.test.ts`) is a routine reviewed change authorized WITHIN this suite once AC-S39-1..8 are green; it is explicitly not held "pending". The only owner-run step is the standing operational deploy (`npm run deploy`), which is not a governance blocker.

**Open questions & assumptions.**

- _Answered 2026-07-23 (D-AUTOMATION-LINE):_ automated notifications to internal staff are authorized; every client-facing send stays human exact-confirmed (roadmap §3, AGENTS.md Roadmap Build Authorization). This suite is internal-only automation.
- _Assumption:_ "follow-up due" means a report `new` for more than one business day, or `acknowledged` (not `resolved`) for more than three business days, computed on read from status + `created_at`; no new Firestore index or collection. Owner-tunable; recorded as a `Q-`/`A-` row in `docs/facts.md` at build time.
- _Assumption:_ the internal transactional sender identity is the existing internal transactional sender (the `KB_APPROVAL_SENDER` internal path) over the already-approved Gmail send scope, server-side; no NEW external scope, credential, or endpoint. This is why S39 appears on no line of roadmap §5.
- _Assumption:_ the transactional email payload is metadata-only (route pathname, origin, reporter role, ISO time, deep link to `/admin`), never the free-text description or element hint, preserving the `F-SUPP-1` / TIX-8 privacy invariant.
- _Assumption:_ support signals are standing (no `read_at`), Admin-scoped, and do not count toward the bell unread-EVENT badge, consistent with how S17 treats connections/coverage/team_review.
- _Open:_ whether internal auto-send also fires on "follow-up due" as a daily standing digest, or ONLY on "new feedback filed". Default for V1: send on new-filed only; follow-up-due stays in-app. A daily digest would require Cloud Scheduler, which this suite does not add (mirrors S17 G3). Recorded as a `Q-` row at build time.
- _Assumption:_ the gate flip is a routine reviewed change inside this suite (not an owner dependency); the only owner-run step is the standing deploy. Decision-complete: a builder can implement all of B1 to B5 and the seam executor with no further owner question.

Suite-specific hard stop: the transactional lane must NEVER resolve a recipient from anything other than the owner-configured internal destination, that destination MUST pass a code-enforced internal-domain allowlist (enforced at config-set AND re-asserted at send), and it is resolved through a non-actor-gated SYSTEM read (never the Admin-gated actor read, which would 403 the non-Admin reporters who file most feedback); a caller-supplied recipient, a non-internal destination, or a client recipient reaching a send is itself a falsification.
