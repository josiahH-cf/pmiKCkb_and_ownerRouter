# S32 - KB corrections learning loop + source freshness + model-config (app-plane)

> New 2026-07-23 (operator note). Wave-1 pure app-plane suite from `docs/roadmap-unblock-2026-07-23.md` (matrix rows #6 "Corrections improve the AI" and #14 "newer KB models"). It has NO live provider and NO owner flip: every slice stays `production_allowed:false`. The disposable decision-complete packet lives at `docs/temp/kb-corrections-learning-plan.md` (local-only); this is the TRACKED spec the loop executes.

**Goal.** Today a wrong or thin answer dies on the screen: the Ask result (`components/ask/AskForm.tsx`) shows the answer, its sources, and an "Answered by <model>" line, but the operator cannot tell the app it got something wrong, and the ask log writer persists no correction at all (`lib/firestore/ask-logs.ts` writes `ask_logs` with no feedback field, even though `AskLogRecord.user_feedback?` exists unused at `lib/firestore/types.ts:319`). Separately, a citation can show "reviewed <date>" but never says whether that review is overdue, and the answer model is configuration nobody can see in-app. After this suite the operator can, on any answer, file a plain-language correction that is captured as a Proposed record and changes NOTHING on its own; an Admin reviews Proposed corrections and, on approval, a deterministic pipeline turns each into a proposed KB entry (a Draft placeholder through the existing editable-layer approval path), a redaction-required eval-case proposal, and a per-source re-rank hint, exactly mirroring the proven Gmail triage loop where "Dan's corrections become Proposed rule changes that require Admin approval; nothing self-modifies" (`lib/gmail-inbox-zero/rules.ts:11`). Citations gain an honest freshness signal (a source review interval plus a computed "review due" or "stale" chip), and Admins get a read-only model-config panel that shows which answer/classify model and provider are live. The honest framing is that corrections HARDEN the app (they feed reviewed KB entries, eval cases, and re-rank hints); nothing here trains or fine-tunes a model, and no correction ever self-applies.

**What it is / how it functions.** One capture surface on the Ask result feeds a Proposed-only record; a pure proposal pipeline (mirroring `rules.ts`) turns Admin-APPROVED corrections into human-reviewed KB/eval/re-rank proposals through the existing editable-layer + approval-queue paths; a pure freshness calculator stamps a staleness signal onto citations; and a read-only admin panel surfaces the answer model. No new external scope, no system-of-record write, no autonomous send.

- **Capture surface - `components/ask/AskForm.tsx` + new `POST /api/ask/correct`.** A "Suggest a correction" control sits beside the existing Capture Task control in the result panel (same `capturableStates` neighborhood), opening a small form: what was wrong (enum: wrong fact, wrong source, missing detail, wrong process) plus a free-text corrected note. It posts to a new route backed by a new `lib/firestore/ask-corrections.ts` writer that appends one `ask_corrections` record with `status: "Proposed"`, the acting `user_uid`, the referenced `ask_log` id, the question, and the returned `source_state`/citations for context. It mirrors the shape and validation of `AskCaptureRequestSchema` (`lib/schemas.ts:46`). Capturing a correction NEVER re-runs the answer and never mutates the answer, citations, KB, or model. The latent `AskLogRecord.user_feedback?` field is the minimal precedent; the dedicated collection is the richer, reviewable form.
- **Proposal pipeline - new `lib/kb-corrections/propose.ts` (pure, mirrors `rules.ts`).** Pure, non-executable, no I/O, no `Date.now()`, exactly like `proposeRuleChangeFromFeedback` (`lib/gmail-inbox-zero/rules.ts:128`). Given one correction it deterministically produces three `status: "Proposed"` artifacts: (1) `proposeKbEntryFromCorrection` -> a Draft placeholder shape for the editable layer (`lib/firestore/editable.ts` `createPlaceholder`, which already enforces its own Draft -> Approved lifecycle, RBAC via `assertCan`, and change-log); (2) `proposeEvalCaseFromCorrection` -> a `redaction_required: true` eval-case proposal that is NEVER written to a tracked `tests/eval` fixture by the app; (3) `proposeRerankSignalFromCorrection` -> a per-`source_id` down/up weight HINT that only a future human-approved config could consume. Every artifact carries the originating correction id and requires Admin approval before it is anything more than a proposal.
- **Review lane - new `components/admin/KbCorrectionsPanel.tsx` in `app/admin/page.tsx` (App Info and Readiness section).** Admin-only (the page is already `requirePageCapability("manageAdmin")`). Lists Proposed corrections with Approve / Dismiss. Approve is the ONLY path that acts: it calls the existing editable-layer `createPlaceholder` to file the proposed KB entry as a Draft in the target Space (which then follows its own approval-queue lifecycle) and marks the correction `Approved`; Dismiss marks it `Dismissed`. There is no control anywhere that turns a correction directly into an Approved KB entry, an active re-rank weight, or a tracked eval fixture. This mirrors the Gmail governance surface (`app/admin/gmail-inbox-zero/page.tsx`).
- **Source freshness - new `lib/retrieval/source-freshness.ts` + `SourceMetaRecord.review_interval_days?`.** A pure `computeSourceFreshness({ lastReviewedAt, reviewIntervalDays, referenceDateIso })` returns `{ status: "fresh" | "review-due" | "stale" | "unknown", dueDateIso?, daysOverdue? }`. `unknown` when either input is absent (honest absence, never a fabricated date). `withSourceMeta` (`lib/retrieval/vertex-search.ts:382`) stamps the result onto the citation next to the existing `last_reviewed_at` surfacing (Slice 5, D13), via a new optional `CitationSchema` field. `AskForm` renders a "Review due" or "Stale" chip only when computed, beside the existing "reviewed <date>". The interval is a new optional `SourceMetaRecord` field (`lib/firestore/types.ts:299`) set through the existing source-meta path (`npm run seed:source-meta`) plus a small Admin control; absent interval = `unknown` = no chip.
- **Model-config surface - new `components/admin/ModelConfigPanel.tsx`.** Read-only. Reads the already-available `config` in `app/admin/page.tsx` and shows the answer model (`config.geminiAnswerModel`, default `gemini-2.5-pro` per `lib/config/server.ts:65`), the classify model (`config.geminiClassifyModel`, default `gemini-2.5-flash`), and the provider (`config.modelProvider`), each via `friendlyModelLabel`. An id in the known-good map renders its friendly label; an unknown id renders the title-cased fallback plus a "not in the known-good list" note. The panel exposes NO runtime mutation control: swapping to a newer KB model is an env change to `GEMINI_MODEL_ANSWER` plus an owner-run deploy, which the panel states plainly. This is the "newer KB models" half of roadmap row #14; the budget-alert WORDING half is S39.

- **Buildable now (app-plane).** The whole suite is app-plane. Every slice adds no external scope, no system-of-record write, no autonomous send, and stays `production_allowed:false`.
  - **Slice 1 - correction capture.** `lib/firestore/ask-corrections.ts` (append-only `ask_corrections`, `status: "Proposed"`), `POST /api/ask/correct`, the "Suggest a correction" control in `AskForm`, and the `CorrectionRequest` schema in `lib/schemas.ts`.   - **Slice 2 - pure proposal pipeline.** `lib/kb-corrections/propose.ts` (pure `proposeKbEntryFromCorrection` / `proposeEvalCaseFromCorrection` / `proposeRerankSignalFromCorrection`, all `status: "Proposed"`, deterministic, no I/O). New `tests/unit/kb-corrections-propose.test.ts` (determinism, Proposed-only, no self-approve).
  - **Slice 3 - Admin review lane.** `components/admin/KbCorrectionsPanel.tsx` wired into `app/admin/page.tsx`; Approve routes through the existing `createPlaceholder` (Draft) + marks `Approved`; Dismiss marks `Dismissed`. New `tests/unit/kb-corrections-panel.test.tsx`.
  - **Slice 4 - source freshness.** `lib/retrieval/source-freshness.ts` (pure), `SourceMetaRecord.review_interval_days?`, the new optional `CitationSchema` freshness field, `withSourceMeta` wiring, the `AskForm` chip, and a small Admin interval control. - **Build to the seam (live provider).** None. This suite has no live provider, no fake/synthetic provider to replace, and no preview/confirm/receipt/rollback external contract. It writes only in-boundary Firestore records (`ask_corrections`) and reuses the existing editable-layer writes, which are already app-plane. Say so explicitly: there is nothing to build to a seam here.

This suite neither performs nor requires either; it only surfaces and proposes.

**Forbidden actions / hard gates.** App-plane only; a violation of any of these is itself a falsification. This suite MAY NOT set `production_allowed:true` for anything (it adds no Action Registry entry).
