# S35 - LeadSimple connector activation

> New 2026-07-23 (operator note). Wave-2 seam spec for the full-suite build program (`docs/roadmap-unblock-2026-07-23.md`, feature #9, §4 Wave-2, §5 owner-dependency #5); authorized by `F-ROADMAP-BUILD-AUTHORIZED`. The disposable decision-complete packet is `docs/temp/leadsimple-activation-plan.md` (local-only); this is the TRACKED spec the loop executes.

**Goal.** Today LeadSimple is a named connector with a fully built executor that can only ever touch a fake. The `LeadSimpleMaintenanceExecutor` (`lib/maintenance/execution/providers.ts:624`) already does the correct governed dance for both LeadSimple actions the app knows about (advance a renewal-pipeline process stage, create a follow-up task): stage-drift check, conditional compare-and-set update, post-write readback, and idempotent reconcile. The one thing missing is a real REST client behind the `LeadSimpleProvider` seam (`providers.ts:603`); the only provider that exists is the synthetic fake wired in `lib/release/synthetic-execution.ts:849`. After this suite the app carries a live `LeadSimpleProvider` (a real REST client) wired into a live executor map that plugs into the same `ExternalExecutionOrchestrator` the Test lane uses, so the moment the owner supplies the one missing dependency the two actions can run for real behind the existing preview, confirm, receipt, and rollback contract. What stays owner-gated is narrow and exact: both `leadsimple.*` registry entries carry `evidence_status:"Vendor-Confirmation-Required"` (`lib/integrations/action-registry-seed.ts:203,246`), so they are DOUBLY gated. The schema refuses `production_allowed:true` unless `evidence_status` is `"Documented"` (`lib/firestore/schemas.ts:311`), and the evidence cannot honestly become `Documented` until the vendor confirms the endpoint-by-endpoint contract. The loop builds everything up to that vendor confirmation and stops there. It never guesses a LeadSimple endpoint and never flips the gate on a hunch.

**What it is / how it functions.** One live provider implementation fills the existing seam; the executor, registry entries, preview/confirm/receipt/rollback path, and lane guards are already built and unchanged. The work splits into three explicit lists so the loop builds up to the single owner step.

- **Buildable now (app-plane).** State and pin what already exists so it cannot silently regress. The `LeadSimpleMaintenanceExecutor` (`providers.ts:624`) implements `ExternalExecutor` for `leadsimple.process.update_stage` and `leadsimple.task.create`; it validates required fields (process id, current and target stage, task ref, title, assignee, ISO due date), enforces that target stage differs from current, reads the process first and refuses on stage drift, applies a conditional `updateStage` keyed by the idempotency key, refuses when `applied` is false or the returned process ref differs, reads back and returns `ambiguous` unless `leadSimpleStageMatches` / `leadSimpleTaskMatches` holds, and reconciles by idempotency key (`reconcileStage` / `reconcileTask`). The two registry entries stay `readiness:"Needs Connection"`, `evidence_status:"Vendor-Confirmation-Required"`, `production_allowed:false`, with the honest `documented_evidence` that LeadSimple exposes an admin-enabled REST API and a direct RentVine integration but that endpoint coverage sits behind authentication. The connector catalog entry (`lib/connections/connector-catalog.ts:76`: id `leadsimple`, method `api_key`, `requiredConfig:["LEADSIMPLE_API_KEY"]`, `healthCheckRef:"health.leadsimple.rest_api"`) stays presence-only. No new external scope, no send, no system-of-record write, all `production_allowed:false`. The loop builds these unattended.
- **Build to the seam (live provider).** A new `LiveLeadSimpleProvider implements LeadSimpleProvider` (new `lib/maintenance/execution/live-leadsimple-provider.ts`), modeled exactly on the sibling `LiveRenewalGmailDraftProvider` (`lib/lease-renewal/execution/live-gmail-draft-provider.ts`): it takes a narrow injected REST client interface (`LeadSimpleRestClient` with `readProcess` / `updateStage` / `createTask` / `readTask` methods), so unit tests inject a fake and no test ever contacts LeadSimple. It maps the six `LeadSimpleProvider` methods onto that client, passes the idempotency key through on every mutating call, and preserves the compare-and-set semantics (`expectedStageRef` -> `targetStageRef`, `applied` reflects whether the conditional write took). The base URL and the per-action paths are NOT literals in the module; they are read from configuration that is absent by default, so with no vendor-confirmed contract the provider constructs into a fail-closed state and emits no network call. Then a live executor map (new `lib/maintenance/execution/live-executors.ts`, the production twin of the synthetic `maintenanceExecutors` map at `synthetic-execution.ts:850`) constructs `new LeadSimpleMaintenanceExecutor(new LiveLeadSimpleProvider(client))` under both `leadsimple.*` keys and hands the map to the same `ExternalExecutionOrchestrator` (`lib/external-execution/orchestrator.ts:136`). Unlike the synthetic map these executors are NOT branded with `markIsolatedTestExecutor`, so the orchestrator lane guard treats them as live; the orchestrator still refuses to run any key whose registry entry is `production_allowed:false`, which both `leadsimple.*` keys remain until the flip. The loop builds ALL of this plus its fake-client Test proof. It does NOT stop merely because the action is external; it stops only at the one owner dependency below.
- **Owner dependency (the one flip).** A single owner step from `docs/roadmap-unblock-2026-07-23.md` §5 item #5: a LeadSimple admin-enabled REST API key (into `LEADSIMPLE_API_KEY` via Secret Manager, never git) AND a vendor-confirmed endpoint contract (the exact stage-update and task-create request/response shape and paths). That vendor confirmation IS the documentation that upgrades `evidence_status` from `Vendor-Confirmation-Required` to `Documented`. Once it lands, the flip is one reviewed change per the roadmap §6 recipe: set both `leadsimple.*` seed entries to `readiness:"Approved for Execution"` + `evidence_status:"Documented"` + `production_allowed:true`, add both keys to the TWO `EXECUTABLE_ALLOWLIST` copies (`scripts/seed-action-registry.ts:31` and `lib/admin/migration-readiness.ts:156`), record the confirmed paths in the provider config, and update the pinned tests (`tests/unit/action-registry-schema.test.ts`, `tests/unit/seed-action-registry-allowlist.test.ts`). The loop hands back only here.

**Open questions & assumptions.**

They are the only `leadsimple.*` keys seeded (`action-registry-seed.ts:196-257`) and the only ones the executor handles. Any third action (for example a note or a webhook subscription) is out of scope until separately specced. Recorded as the S35 boundary.

**Forbidden actions / hard gates.** Doubly gated by construction: no flip is possible while `evidence_status` is `Vendor-Confirmation-Required`, because the schema (`lib/firestore/schemas.ts:311`) refuses `production_allowed:true` without `Documented` evidence, and the evidence cannot honestly become `Documented` until the vendor confirms the endpoint contract. No guessed LeadSimple endpoint, host, or path ever enters source, git, or evidence; the live provider fail-closes without a vendor-confirmed contract. Every live effect stays one-attempt, idempotent (idempotency-key keyed), read-back verified, reconcilable, and reversible (stage rollback = set the recorded prior stage; task rollback = delete or close the created task), exactly as the executor already enforces. Human confirmation stays mandatory: each run passes the S26 preview/confirm/receipt path; this suite adds no autonomous client-facing send and no generic `gmail.message.send` (that stays Registry-closed). No personal `josiah.abernathy@gmail.com` in any auth path. No secrets or customer PII in git; `LEADSIMPLE_API_KEY` loads from env / Secret Manager. `~$10` total cost ceiling under the real kill switch. This suite MAY build the live provider and the live executor map and prepare the flip; it does NOT set `production_allowed:true` until the named owner dependency is documented, at which point the flip updates both `EXECUTABLE_ALLOWLIST` copies plus the pinned schema tests. A violation of any of these is itself a falsification.
