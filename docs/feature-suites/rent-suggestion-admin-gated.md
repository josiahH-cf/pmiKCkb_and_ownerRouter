# S29 - Comp-informed rent suggestion, Admin-approval-gated

> New 2026-07-23 (operator note). Implements owner decision `D-RENT-SUGGEST` (`docs/roadmap-unblock-2026-07-23.md` feature #2, §3, Wave-1) and is the single governance change that opens the former owner-rent hard exclusion. The disposable decision-complete packet lives at `docs/temp/rent-suggestion-admin-gated-plan.md` (local-only). This is the TRACKED spec the loop executes.

**Goal.** Today the app can DRAFT the owner renewal email and capture the operator's own comp basis, but it is forbidden to compute, suggest, or negotiate a rent figure of its own: every number in the draft is the operator's own input, and an absent number renders a visible `Needs Verification` marker (`F-NEGOTIATION-EXCLUDED`, `buildOwnerRenewalDraft`). The owner has now opened exactly one door (`D-RENT-SUGGEST`, 2026-07-23): the app MAY compute a comp-derived SUGGESTED renewal rent number from S28 comp data, but that number does nothing until an Admin explicitly approves that exact figure. After this suite, an Admin working a live lease sees the suggested number rendered next to the comp sources that produced it and a required per-number Approve control; only after that approval does the exact approved number flow into the owner-notice draft as a source-tagged suggestion, and a human still reviews and sends. Absent comp data still renders `Needs Verification`, never a fabricated value; the suggested number always shows its comp sources; and `owner_money` stays a hard exclusion for everything except this one Admin-approved rent-number path. Nothing else about the draft-only, human-sends posture changes.

**What it is / how it functions.** One pure compute module feeds one Admin-gated per-number approval control (an exact clone of the built write-back approval control plane); only an Approved record lets the number cross into the existing owner-draft composer. No system-of-record write, no send, no model call anywhere in the path.

- **Compute - new `lib/lease-renewal/rent-suggestion.ts`.** A pure `computeRentSuggestion({ comps, currentRent, policy })` over the comp set S28 supplies (or, until S28 lands, over the operator's already-captured `RenewalMarketBasis` Zillow low/high plus PMI number). It returns a `RentSuggestion` value: `{ suggestedRent: number | null, method, comps: CompSource[], status: "suggested" | "needs_verification", rationale }`. Deterministic and side-effect free: no `Date.now()`, no I/O, no model call. When the comp set is empty or absent it returns `status: "needs_verification"` with `suggestedRent: null` (never a fabricated number), and it never emits a `suggestedRent` without a non-empty `comps` source list.
- **Approve (pure FSM) - new `lib/lease-renewal/rent-suggestion-approval.ts`.** A byte-for-byte mirror of `lib/lease-renewal/writeback-approval.ts`: states `Awaiting Approval` to `Approved` / `Returned for Revision`, a `planRentSuggestionApprovalDecision(decision, previousState)` that throws `EditableLayerError(409)` on an illegal transition (double-approve, re-return), and every plan carries the compile-checked `productionAllowed: false` invariant. A recomputed different number makes a prior approval stale by construction.
- **Persist (control plane) - new `lib/firestore/lease-renewal-rent-suggestion-approvals.ts`.** A mirror of `lib/firestore/lease-renewal-writeback-approvals.ts`: `assertCan(actor, "manageAdmin")` on every decide, a single-transaction decide that rejects the illegal transitions, an append-only Activity row per decision, and the same stale rule (the record stores the EXACT approved number and its comp sources; a later suggestion with a different value or different sources makes the prior approval stale, so a changed number is never silently authorized). Every record carries `production_allowed: false` and `executed: false`. Keyed by `leaseId`. It NEVER executes a send or a system-of-record write; approving records human authorization to place the number in a draft, nothing more.
- **Endpoint - new `app/api/lease-renewal/rent-suggestion/route.ts`.** POST approve / return, modeled on `app/api/approval-queue/[itemId]/route.ts`: Admin-gated (`manageAdmin`), returns 403 for any lesser role, 200 plus the read-back record for an Admin, maps `EditableLayerError` to its HTTP status.
- **Surface - new `components/lease-renewal/RentSuggestionApproval.tsx`.** Rendered by `components/lease-renewal/RenewalWorkspace.tsx` in the Owner-decision step, beside the existing `OwnerDecisionForm` (`components/lease-renewal/RenewalProgressControls.tsx`). It shows the suggested number ALWAYS adjacent to its comp sources, the current approval state, and the required per-number Approve / Return control. The control is Admin-only (`can(user.role, "manageAdmin")`); a non-Admin sees the number and its sources read-only with no approve affordance. A `needs_verification` suggestion renders the `Needs Verification` text and no number.
- **Draft flow - modify `lib/lease-renewal/owner-draft.ts` (`buildOwnerRenewalDraft`).** `OwnerDraftMarketInput` gains an approved-suggestion channel. When (and only when) an `Approved` rent-suggestion record exists for the lease, its exact value populates the existing "Suggested market value" fact and the `suggestion_line`, tagged with a distinct source label ("Comp-derived suggestion (Admin-approved)") plus its comp sources, so the draft is transparent about where the number came from. With no `Approved` record the behavior is unchanged: the operator's own PMI number, or the `Needs Verification` marker. The computed number NEVER auto-populates the draft; only the Approved record does.
- **Provenance - modify `lib/lease-renewal/renewal-progress.ts` (`RenewalMarketBasis`).** Optionally carry the approved-suggestion provenance (the approval record id and its comp sources) so the recorded comp basis and the composed draft agree on exactly which approved number was used, and the write-back comp-basis proposal (`buildCompBasisProposal`) can cite it.
- **Gate narrowing - modify `lib/gmail-inbox-zero/draft-safety.ts`.** `inspectGmailDraftSafety` stays deterministic, fail-closed, and model-free. `GmailDraftSafetyInput` gains an optional, SERVER-SET `approvedRentSuggestion` provenance (never client-trusted; resolved server-side from the control plane above). A single narrow branch allows an owner-renewal draft whose only owner-money-class signal is the Admin-approved renewal rent number; the existing `owner_money` category exclusion and the `detectExcludedIntent` owner-money regex are left BYTE-FOR-BYTE unchanged for every other input, so payout / funds / monies / proceeds / payment content stays hard-excluded and any draft lacking a valid Approved record is refused exactly as today.

- **Buildable now (app-plane).** The WHOLE suite ships here. No new system-of-record write, no autonomous send, no new external scope; `production_allowed` stays `false` (the draft is draft-only regardless). The loop builds all of it unattended.

  - Slice 3 - `components/lease-renewal/RentSuggestionApproval.tsx` wired into `RenewalWorkspace.tsx`; number always with its comp sources; Admin-only approve control + `tests/unit/rent-suggestion-approval-component.test.tsx`.
