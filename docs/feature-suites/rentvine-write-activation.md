# S30 - RentVine renewal-write activation

> New 2026-07-23 (operator note). Wave-2 seam suite from `docs/roadmap-unblock-2026-07-23.md` (feature #4, section 3 `D-RENTVINE-ENDPOINT`, section 4 Wave-2, section 5 owner-dependency #1). The disposable decision-complete packet is `docs/temp/rentvine-write-activation-plan.md` (local-only); this is the tracked spec the loop executes. It builds the write half of `F-RENTVINE-WRITE-APPROVED` (the open D16 endpoint) under the build-to-seam rule of `F-ROADMAP-BUILD-AUTHORIZED`.

**Goal.** Today the renewal write-back to RentVine looks finished but is not connected to anything real. The executor, the S25 preview/confirm/receipt/rollback contract, the High-risk approval-queue path, and the one-attempt/idempotent orchestrator all exist and are unit-proven, but the only provider ever wired behind them is an in-memory fake in the Test path (`lib/release/synthetic-execution.ts:524`), and RentVine itself is GET-only (a live read on 2026-07-22 confirmed no write endpoint). After this suite there is a real RentVine HTTP write client behind the same least-privilege `RenewalMutationProvider` seam, constructed by a live executor map (the production path has none today), exercised end to end in the Test lane against a RentVine-shaped fixture, and staged so that activation is a single reviewed change the moment the owner provides the documented RentVine renewal-write endpoint. Nothing writes to RentVine in this cycle: the action stays `production_allowed:false`, the live provider fails closed until the documented endpoint is configured, no endpoint path is ever guessed, and every eventual write is one human-confirmed approval-queue action. The owner holds exactly one step, and flipping the gate after it is a one-line change.

**What it is / how it functions.** One seam, one live client, one staged flip. The write travels the already-built contract; this suite only replaces the fake provider with a real one and prepares the gate, stopping at the single named owner dependency.

- **Executor and contract (real modules, already wired).** `RentvineRenewalExecutor` (`lib/lease-renewal/execution/providers.ts:385`) over the `RenewalMutationProvider` seam (`:370`) already does read-drift-check, compare-and-set with an idempotency key, field-by-field readback, and reconcile: `execute` reads the record, refuses if `lease_ref` or `current_rent` drifted, calls `compareAndSetRenewal({recordRef, expectedLeaseRef, expectedCurrentRent, values, idempotencyKey})`, throws if `applied` is false, re-reads the returned `providerRef`, and verifies `new_rent`/`effective_date`/`lease_end_date`/`fee_cents` before returning a receipt; `reconcile` re-finds the write by idempotency key and matches the same fields. It is unit-proven (`tests/unit/rentvine-renewal-executor.test.ts`).
- **Preview and gate surfaces (built, gated OFF).** The S25 bodyless preview schema `FINAL_V1_ACTION_PREVIEW_SCHEMAS["rentvine.lease.renewal_writeback"]` (`lib/integrations/final-v1-action-contracts.ts:70`) fixes the six fields the approver sees (`lease_ref`, `current_rent`, `new_rent`, `effective_date`, `lease_end_date`, `fee_cents`). The registry entry (`lib/integrations/action-registry-seed.ts:940`) is `readiness:"Planned"`, `evidence_status:"Undocumented"`, `production_allowed:false`. The risk matrix (`lib/lease-renewal/execution/matrix.ts`, entry index 9) is `High` with `requiredContract:"undocumented"`, which the orchestrator maps to a hard "Blocked: vendor contract required." until a documented contract exists.
- **Human-confirm path (built).** The write is `High` / `system_of_record_write` in the risk floor (`lib/execution/risk-policy.ts:51`), which raises `risk_signals.external_write:true` (`lib/execution/service.ts:166`) and routes the action into the human approval queue (`lib/firestore/approval-queue.ts:110`). The orchestrator (`lib/external-execution/orchestrator.ts`) enforces one claimed attempt, duplicate-returns the stored receipt, and requires a matching confirmed preview hash before `executor.execute` runs; production internal actors must go through the S20 execution ledger (`assertProductionDirectActor`).

- **Buildable now (app-plane).** Nothing new is built here: the app-plane and the S25 contract are already shipped and stay exactly as they are. This bucket is the fixed contract the live provider must satisfy, not work to redo. The loop does NOT rebuild the executor, the preview schema, the risk/approval routing, or the orchestrator; it treats them as frozen and keeps their sentinels green (`rentvine-renewal-executor.test.ts`, `execution-risk-policy.test.ts`, `execution-approval-queue.test.ts`, `action-gate.test.ts`). The registry entry stays `production_allowed:false` and the key stays off both allowlists this cycle.
- **Build to the seam (live provider).** The real new work, all up to but NOT including the one-line flip:
  - **Live write transport - `lib/integrations/rentvine/` (new write surface).** The existing client (`lib/integrations/rentvine/client.ts`) is read-only and its `RentVineHttpTransport` hard-codes `method:"GET"`; it stays that way. Add a separate, clearly named write surface (for example `client-write.ts`) that keeps the same guarantees: HTTP Basic auth built lazily and never logged or thrown, the `pmikcmetro` host guard (`assertRentVineAccount`), the https-only base-URL parse, and the `AbortController` timeout. The write verb and path are NEVER literals; they are supplied only from the documented-endpoint config value.
  - **Live `RenewalMutationProvider` - new `lib/lease-renewal/execution/rentvine-mutation-provider.ts`.** Implements `compareAndSetRenewal` / `read` / `findByIdempotencyKey` against the write surface, preserving the executor's conditional-update contract. If the documented endpoint is unconditional, the provider composes the same guarantee inside its one allowed attempt (read-verify current state, then write, then let the executor's readback drift-check confirm), so the compare-and-set semantics the executor already enforces are never weakened.
  - **Config - extend `lib/lease-renewal/live-config.ts`.** A `buildLiveRenewalWriteConfig(env)` that requires the documented write endpoint value (for example `RENTVINE_RENEWAL_WRITE_ENDPOINT` plus its semantics) in addition to `RENTVINE_API_BASE_URL`/`KEY`/`SECRET`, account-guarded to `pmikcmetro`, and returns `{ok:false, reason:"not_configured"}` when the endpoint value is absent. The seam is inert until the owner lands the endpoint.
  - **Live executor map - new factory (sibling of the synthetic one).** A production live executor map that constructs `new RentvineRenewalExecutor(liveRenewalProvider)` for `rentvine.lease.renewal_writeback` and is consumed by the S20 execution ledger path, NOT branded as an isolated Test executor. Today only `createIsolatedTestExternalActionOrchestrator` exists and only the synthetic path builds a `leaseExecutors` map. When the write config is `not_configured`, the map offers no renewal-write executor, so preparation blocks with "Provider adapter is unavailable."
  - **Health and preflight.** Reuse `health.rentvine.api_key` (`lib/integrations/rentvine/health-probe.ts`) and add a write-preflight that reports the endpoint as documented-and-configured before the live provider is offered; an undocumented endpoint reports not-ready rather than attempting a call.
  - **Test-lane proof - new `tests/unit/rentvine-mutation-provider.test.ts`.** Drive the real provider class (not the in-memory fake) through a RentVine-shaped fixture transport: prove compare-and-set applies once, an idempotent replay returns the same `providerRef` with no second write, a drifted `current_rent` throws before any write, an ambiguous result stays reconcilable, and `reconcile` matches only when the readback equals the approved values. This never contacts live RentVine.
- **Owner dependency (the one flip).** The single owner step is the documented RentVine renewal-write endpoint plus its semantics (verb, path, request and response shape, whether it supports conditional update and an idempotency key): `D-RENTVINE-ENDPOINT`, roadmap section 5 owner-dependency #1 (owner providing). It is the documentation the gate has been waiting on. Once provided, the flip is one reviewed change: set the documented endpoint value in env or Secret Manager (owner-run); in `lib/integrations/action-registry-seed.ts:940` set `evidence_status:"Documented"`, `readiness:"Approved for Execution"`, `production_allowed:true`; in `lib/lease-renewal/execution/matrix.ts` move the entry's `requiredContract` from `"undocumented"` to `"documented"` and route with a `contractRef` starting `documented:` (so `validateExternalReadiness` stops blocking); add `rentvine.lease.renewal_writeback` to BOTH `EXECUTABLE_ALLOWLIST` copies (`scripts/seed-action-registry.ts:31` and `lib/admin/migration-readiness.ts:156`); and update the pinned tests (`tests/unit/action-registry-schema.test.ts`, `tests/unit/seed-action-registry-allowlist.test.ts`, and the executor proof `tests/unit/rentvine-renewal-executor.test.ts`). Deploy stays owner-run.

**Open questions & assumptions.**

- _Open / Owner-dependency:_ the documented RentVine write endpoint and semantics (verb, path, request/response shape, conditional-update support, idempotency support). Owner providing (`D-RENTVINE-ENDPOINT`, roadmap section 5 #1). This is the single blocker; it is already tracked as the open D16 write half of `F-RENTVINE-WRITE-APPROVED`, so no new `Q-`/`A-` row is needed (facts.md is not edited by this suite). Until it lands, `buildLiveRenewalWriteConfig` returns `not_configured` and no path is guessed.
- _Assumption:_ the documented endpoint reuses the existing RentVine Basic-auth credential as-is (per the RentVine-credential ruling: use, do not rotate) and the same `pmikcmetro` account guard. If the write requires a distinct scope or key, obtaining it becomes part of the owner step, not new app work. Confirm-with-default: reuse the current credential.
- _Assumption:_ if the documented endpoint is a plain unconditional update rather than a native compare-and-set, the provider preserves the executor's guarantee via a read-verify-write inside the one allowed attempt plus the existing readback drift-check. Confirm-with-default: single conditional write modeled on the current executor contract.
- _Assumption:_ the RentVine fields written match the six S25 preview fields (`new_rent`, `effective_date`, `lease_end_date`, `fee_cents` against `lease_ref`/`current_rent`). Owner or Dan verifies the exact RentVine field names before the first live write, per `F-V1-REMEDIATION-DECISIONS`. This suite does not send a live write, so field verification is an owner-step gate on the flip, not a build blocker.
- _Open / Client-owned:_ whether a renewal write is one call or a documented sequence (for example update the lease and create a charge). The provider models the documented shape once known; default is a single compare-and-set matching the executor today.
- _Assumption:_ no autonomous or scheduled write. Every write is one human-confirmed approval-queue action per `D-AUTOMATION-LINE` (system-of-record and client-facing effects stay human-confirmed). Nothing in this suite relaxes that.

**Forbidden actions / hard gates.** No live RentVine write happens in this suite. No guessed endpoint: RentVine is GET-only until the owner provides the documented write endpoint, and the write path/verb is only ever injected config, never a literal. Every write is human-confirmed through the approval queue (`external_write:true`); there is no autonomous, scheduled, bulk, or model-triggered write (`D-AUTOMATION-LINE`, roadmap section 7). Every live effect stays target-labeled, one-attempt, idempotent, readback-verified, reconcilable, and reversible via the documented correction contract. The read client stays read-only. The credential is reused as-is, never rotated, never in git or evidence, loaded only from env or Secret Manager; the account guard stays `pmikcmetro`; the personal account never enters any auth path. This suite MAY build the live provider and the system-of-record write to the seam and prepare its gate flip, but it does NOT set `production_allowed:true` or add the key to either allowlist until `D-RENTVINE-ENDPOINT` is documented; the flip then updates both `EXECUTABLE_ALLOWLIST` copies plus the pinned schema tests in one reviewed change. Deploys and the endpoint/credential set stay owner-run. The ~$10 budget cap holds. A violation of any of these is itself a falsification.
