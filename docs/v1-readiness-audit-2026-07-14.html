Executive conclusion

The app is not broadly blocked.

The core internal operations application is built and heavily verified. Several old blocker statements were stale: the client cloud project, billing, domain auth, Rentvine read, renewal-Sheet read, signed-lease location, renewal walkthrough, Gmail scopes, and Gmail transport have already been resolved or proven. The remaining release gates are concentrated in owner decisions, approved launch content, operating ownership, and final acceptance.

Recommended V1 contract Launch the internal app with app-plane decisions, source-backed Ask, read integrations, test/simulation runs, Maintenance tickets, and workflow-linked communication. Keep every non-Gmail external system mutation behind a later per-action approval. Existing plans already say production launch should precede those write paths.

What “current” means

Two release states must not be conflated

Production baseline

Deployed and transport-proven

The client-owned Cloud Run service is live. Revision pmi-kc-kb-demo-00020-24d proved per-user Gmail DWD, four scopes, synthetic self-send/reply, labels, authenticated push/watch, cursor, and bodyless audit. Earlier production checks covered auth, cited Ask, Console/Notifications, renewal review, Maintenance readiness, and responsive layouts.

Local release candidate

Verified, but not committed or deployed

The July 14 worktree removes the general inbox/compose product surface and replaces it with workflow-linked renewal/maintenance communication, strict entity authorization, bodyless links, governed labels/drafts, exact-confirmed linked replies, targeted attention, and on-demand “Needs Review” analysis. Full local verification passed.

Audit rule: a feature can be built, tested, transport-proven, and still not be deployed or product-authorized. This packet labels those states separately.

Product surface map

What each tab does, how we know, and what remains

“Works” means the intended app-plane behavior exists and has code/test or direct-browser evidence. It does not imply that a gated external action is enabled.

Console · / and /ask

Built and verified
PurposeAt-a-glance front door: decisions, connection/setup gaps, grounded Ask/Dictate, anticipated work, process status, and safe test-run starts.
What worksOne value-free decision gather feeds Console/Queue/Notifications; cited answers and no-source behavior are enforced; process starts are simulation-only.
Known gap / tune-upAnticipated work still uses deterministic sample renewal data. A stale code comment says Console never approves even though safe inline app-plane approval exists.

Evidence: ConsoleView · console tests · production smoke in status

Spaces · /spaces

Built; content-gated
PurposeDirectory of business areas. Each card shows real process/connection readiness and opens its operator desk or reference space. Processes live under Spaces, not as a peer tab.
What worksCards are clickable, role/scope-filtered, status-colored with text, and share the renewal waiting count with the Approval Queue.
Known gap / tune-upTwelve launch-space records exist, while the planned first release names four. Several remain placeholders or lack approved production sources/process definitions.

Evidence: Spaces page · Space registry · state tests

Approval Queue · /approval-queue

Built and verified
PurposeUrgent-first, value-free triage for decisions. Safe Low/Medium queue items may approve inline; renewal flags and writeback proposals deep-link to value-bearing run pages.
What worksPermissions, self-approval prevention, mobile decider, activity, bulk app-plane decisions, renewal review, writeback authorization, metrics, and graceful errors are tested.
Known gap / tune-upExternal execution remains intentionally off. Operational usefulness depends on real queue-producing workflows rather than demo/test records.

Evidence: Queue page · Queue UI · component tests

Communications · /gmail-hub

Local candidate only
PurposeConnection/watch health and bodyless attention for Gmail threads deliberately linked to renewal or Maintenance workflows. Detailed reading/action starts from the workflow entity.
What worksPer-user transport, strict context authorization, link/read, four governed labels, targeted reply attention, on-demand review proposal, route-level draft/reply confirmation, idempotency, and audit.
Known gap / tune-upNot deployed. New links need retention TTL. Renewal initiation is preview-only; Maintenance notice needs trigger/contact/template. The workflow panel currently exposes link/read/label/analyze, not an end-user draft/reply sequence.

Evidence: Communications home · workflow panel · service tests

Connections · /connections

Built and read-only
PurposeShared truth surface for systems the app reads or may later act through. Everyone sees status; Admins receive setup/refresh controls. Secret values never render.
What worksConfiguration presence plus cached read-only verification; Rentvine and Sheets/Drive seams have live evidence; connector cards distinguish connected/action/not connected.
Known gap / tune-upThe Action Registry still labels some proven read capabilities “Needs Connection” because registry readiness and connector health are separate models. UI language should distinguish transport health from action permission.

Evidence: Connections page · catalog · status tests

Admin · /admin

Built; copy tune-up
PurposePeople and access, activity/logs, app readiness, migration, communication governance, role/scope changes, and break-glass-aware administration.
What worksAdmin-only guards, role/scope management, domain/last-Admin checks, append-only audit, observability, readiness report, and inactive legacy notification history.
Known gap / tune-upThe migration page still describes all allowlisted actions as “unsent draft only,” which is stale now that linked read/reply/label are allowlisted. Historical recipient-role controls remain editable even though delivery is disabled.

Evidence: Admin page · migration page · readiness tests

Notifications · bell and /notifications

Built and verified
PurposeFull time-ordered attention hub. It is a superset of the Console, not a replacement: decisions, recent events, connection/setup gaps, space coverage, and Admin review digest.
What worksShared decision count, value-free lanes, per-user preferences, mark-read/all-read, scope filtering, Maintenance events, and linked-reply attention.
Known gap / tune-upThe bell badge intentionally counts unread events, not the full decision backlog; this distinction needs to remain clear in copy and acceptance.

Evidence: Notifications page · hub gather · hub tests

Core workflow depth

Lease Renewal

Strong app-plane
  • Built: deterministic ingest/reconcile/flag, live Rentvine + Sheet reads, precedence, mobile/desktop decisions, resolution audit, writeback proposal/authorization.
  • Not built as real operation: production run creation from authoritative candidates, real recipient-backed initiation, any Sheet/Rentvine/Dotloop/LeadSimple write.
  • Audit verdict: do not call it “discovery only”; do not call it end-to-end automation yet.

Maintenance Intake

Strong app-plane
  • Built: typed/voice capture, live unit match, blocker-free ticket creation, public quarantine intake/review, queue, assignment, status, notes, activity, workflow communication panel.
  • Gated: Drive photo execution, Rentvine work-order create/read/update, owner-notice initiation, vendor selection/communication.
  • Audit verdict: useful internal ticket workflow now; external completion is later under D01.

Processes and workflow runs

Framework built
  • Built: Draft/Testing definitions, process pages, simulation-only test-run starts, workflow-run detail, version/readiness concepts.
  • Not yet a production automation engine: test runs do not touch external systems; operational Active definitions need approved sources and action gates.

Source-backed KB

Content gate
  • Built/live-proven: auth, grounded Ask, citations, source-state uncertainty, no-reliable-source behavior, Admin/source tooling.
  • Remaining: approve the launch-Space source scope, sensitivity, corpus manifest, and acceptance questions.
