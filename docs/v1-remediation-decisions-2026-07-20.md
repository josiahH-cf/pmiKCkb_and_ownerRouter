## Governance rulings that change durable direction

Four rulings change durable governance and should be reflected wherever future work reads context:

1. **Onboarding (D1):** universal self-register stays, but a new account lands with no space access until an Admin assigns scope. Introduce an explicit "unprovisioned" state rather than changing the meaning of absent `scopes` (still all-spaces for provisioned users), so existing admins are never locked out.
2. **Send authority (D2):** a single Editor's exact-confirm is the accepted v1 authority to send a workflow-linked tenant or owner email. No maker/checker split for v1. `sendEmail` remains an Editor capability by design.
3. **Activation (D3):** `publish` is the one canonical path to Active; `activate` is retired or redirected.
4. **Templates (D6):** the template store becomes the single editable source of message copy for v1, an in-app editor with an approval lifecycle that every composer reads from. This evolves the existing immutable v1.0-artifact approach and must be reconciled with it during the template-subsystem build.

## Safety invariants (unchanged by any ruling)

No ruling weakens the safety model. The renewal path stays draft-only; no source-of-truth write without approval; the action-registry gate coupling holds; vendor and internal sessions stay isolated; demo auth stays off in production; any real draft's recipient stays server-resolved from the authoritative live source; Test-lane records stay isolated and never reach a real recipient. Each resulting Change task must re-assert its invariant, especially D6's composer wiring and D7's honesty fix (neither may create a new send path) and D1 (must not lock out existing admins).
